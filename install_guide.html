API Crebortoli — Guia de Instalação

Instalação com Docker containers em servidor Debian

Arquitetura Docker

A API Crebortoli agora roda em containers Docker gerenciados pelo Docker Compose. Abaixo a topologia após a instalação:

Host Debian
Nginx proxy reverso · porta 80/443
←→
Porta {{APP_PORT}} 127.0.0.1:3001
↓
Container API Node 20 Alpine · Fastify
←→
Container PostgreSQL Postgres 16 Alpine
uploads/ bind mount · /var/www/crebortoli/uploads
pgdata volume Docker · dados do banco
docker-compose.yml 2 serviços · 1 rede bridge
docker-compose up -d • restart: unless-stopped • healthcheck no db
Fluxo:
Cliente → Nginx (host) → porta {{APP_PORT}} (host) → Container API → Container PostgreSQL
Uploads de arquivos vão para o bind mount /var/www/crebortoli/uploads/, persistindo mesmo após docker compose down.

Passo a Passo

1
Baixar o script

Obtenha o install_crebortoli.sh no servidor Debian.

2
Executar instalação como root 
sudo bash install_crebortoli.sh

O script instala automaticamente:

  • docker.io — Docker Engine
  • docker-compose-plugin — Docker Compose
  • nginx — proxy reverso
3
Configurar variáveis

O script solicitará interativamente:

VariávelPadrãoDescrição
APP_PORT3001Porta no host para a API
DB_NAMEcrebortoli_dbNome do banco PostgreSQL
COMPOSE_PROJECT_NAMEcrebortoliNome do projeto Docker e prefixo dos containers
4
Geração dos artefatos

O script cria toda a estrutura em /var/www/crebortoli/:

ArquivoFunção
.envVariáveis de ambiente lidas pelo docker-compose
DockerfileImagem Node 20 Alpine com a API
docker-compose.ymlOrquestração dos containers (api + db)
api/package.jsonDependências npm (Fastify, pg, etc.)
api/src/server.jsCódigo da API Fastify (~700 linhas)
uploads/Diretório de uploads (bind mount, owner 1000:1000)
migrations/001_create_tables.sqlSQL de referência (tabelas criadas automaticamente no startup)
5
Build da imagem Docker 
docker compose -f /var/www/crebortoli/docker-compose.yml build

Instala as dependências npm dentro do container Alpine e prepara a imagem.

6
Iniciar containers 
docker compose -f /var/www/crebortoli/docker-compose.yml --project-name crebortoli up -d

O Docker Compose inicia na ordem:

  1. Container db (PostgreSQL 16 Alpine) — aguarda health check
  2. Container api (Node 20 Alpine) — conecta ao db, cria tabelas automaticamente, inicia servidor Fastify na porta 3001
7
Nginx configurado

O script gera/configura o nginx para fazer proxy reverso:

https://api.projetosdinamicos.com.br/crebortoli/ → http://127.0.0.1:3001/

E recarrega o nginx automaticamente.

8
Verificação

O script testa o endpoint /health da API e exibe o resultado ao final.

Comandos úteis pós-instalação:
docker compose -f /var/www/crebortoli/docker-compose.yml logs -f — logs em tempo real
docker compose -f /var/www/crebortoli/docker-compose.yml restart — reiniciar serviços
docker compose -f /var/www/crebortoli/docker-compose.yml exec api sh — shell no container api
docker compose -f /var/www/crebortoli/docker-compose.yml down — parar tudo

Migrando banco existente

pg_dump -U postgres crebortoli_db > /tmp/crebortoli_db.sql
docker compose -f /var/www/crebortoli/docker-compose.yml \
  exec -T db psql -U postgres crebortoli_db < /tmp/crebortoli_db.sql

Versão Antiga vs Nova (Docker)

Versão Antiga (PM2)

  • Node.js / npm instalado diretamente no host
  • PM2 gerenciando o processo Node
  • PostgreSQL instalado no host
  • npm install executado no host
  • psql executando migrations no host
  • Dependências do sistema: nodejs, postgresql, pm2
  • Uploads salvos em diretório local no host

Nova Versão (Docker)

  • Node 20 Alpine dentro do container api
  • Docker gerencia restart (policy unless-stopped)
  • PostgreSQL 16 Alpine dentro do container db
  • npm install executado no Dockerfile (cacheado)
  • Migrations automáticas no startup da API (server.js)
  • Dependências do sistema: docker.io, docker-compose-plugin
  • Uploads em bind mount no host (/var/www/crebortoli/uploads)

Vantagens e Desvantagens

Vantagens do Docker

  • Isolamento completo — Node e PostgreSQL não poluem o host
  • Reprodutibilidade — mesma imagem em dev, staging, produção
  • Facilidade de deploy — docker compose up -d resolve tudo
  • Rollback rápido — basta trocar a tag da imagem e recrear
  • Dependências mínimas no host — só Docker e nginx
  • Health check automático — API só sobe após DB estar pronto
  • Escalabilidade horizontal — múltiplas réplicas com --scale
  • Atualização segura — troca a imagem sem afetar outros serviços

Desvantagens do Docker

  • Overhead leve — ~50-100 MB adicionais por container vs nativo
  • Curva de aprendizado — conceitos de Docker, Dockerfile, volumes
  • Acesso a arquivos — uploads em bind mount precisam de permissão explícita (chown 1000:1000)
  • Debug mais complexo — precisa de docker exec ou logs
  • Latência de rede mínima — tráfego passa pela bridge Docker (vs localhost nativo)
  • Manutenção da imagem Docker — builds precisam ser refeitos ao atualizar dependências
  • Docker daemon — mais um serviço rodando no host, consumo extra de RAM
Resumo: Para este cenário (API única em produção), o Docker oferece mais benefícios do que desvantagens. O overhead é mínimo em servidores modernos, e o ganho em isolamento, portabilidade e facilidade de manutenção supera a complexidade adicional.
